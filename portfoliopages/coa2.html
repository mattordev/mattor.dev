Overview

Gather your cards, weapons and potions because the corruption is growing stronger. Push into the heart of the realm and find the source of this terrible plague. Traverse the world in this 2.5D tarot-themed RPG card combat deck builder. Find new items and discover new lands. Anything is possible when fortune favours the bold.

Corruption of Arcana was my third-year project. We worked collaboratively in a team of 12, divided into 5 smaller departments:

  • Programming
  • 2D Art
  • 3D Art
  • Audio
  • Design

I was in the programming department, and my main role was to implement the World AI and provide support as the version control manager.

Technical Breakdown

Unity project split into exploration, combat, and shared progression systems.

  • Console tooling: I built and extended debug commands for faster testing.
  • World AI: I improved enemy state flow and NavMesh roaming behavior.
  • Persistence: I added save/load hooks for scene, player, quest, and card progress.

Console And Tooling

I implemented command parsing for the in-game developer console, including /give and quest/debug commands. This let us test progression quickly without rebuilding scenes.

AI Systems

I contributed to enemy state behavior (wander/chase/follow) and improved roaming logic using NavMesh movement and patrol tooling iterations.

Saving And Progression

I worked on save features for scene/player continuity and progression state so exploration and combat transitions remained consistent.

I also owned Android build delivery for Battle for Arcana and supported version control workflow to keep team integrations stable.

Challenges & Solutions

The two most persistent technical challenges were debug tool reliability in builds and maintaining stable progression continuity across scenes.

Challenge 1 - Developer Console Reliability

The in-game developer console was essential for testing progression, quest states, and item grants. However, in certain builds it behaved inconsistently - sometimes failing to appear, or disabling gameplay UI without properly restoring it.

This slowed iteration significantly, especially when testing edge cases or late-game progression scenarios.

Solution - State Handling & Command Flow Refactor

I refactored the console toggle flow and aligned UI enable/disable logic with explicit state checks. I also improved command parsing reliability for commands such as /give and quest-debug tools.

  • Stabilised UI toggle behaviour in builds
  • Improved debug command consistency
  • Reduced time spent reproducing progression bugs

Challenge 2 - Cross-Scene Progression Continuity

Exploration and combat existed in separate scenes, creating fragile transitions. Player stats, quest flags, and deck progression could reset or desynchronise when returning from combat.

Early issues included:

  • Quest progress not restoring correctly
  • Inventory duplication or card reset bugs
  • Scene return context inconsistencies

Solution - Structured Save Hooks & Data Ownership

I researched and documented the save pipeline before refactoring it. I defined explicit persistence categories:

  • Scene index and return position
  • Player health and stat state
  • Quest progression flags
  • Deck composition and upgrades
  • Inventory item lists

I implemented structured save hooks for each system and added load-time validation checks to prevent corrupted or partial restores.

Item Tracking Improvements

To prevent desynchronisation, I formalised an item save list system that serialised inventory and card data into controlled data models, separating runtime objects from persistent state.

  • Runtime objects rebuilt from saved data on load
  • Reduced duplicate or missing item bugs
  • Improved long-session QA stability

Outcome

The result was faster internal QA cycles, more reliable builds, and a stable cross-scene progression loop that supported extended playtesting without regression.

What I Learned

This project reshaped how I approach system design, debugging tools, and data ownership.

I learned that unreliable internal tools can slow a team down more than missing features. Stabilising the developer console significantly improved iteration speed and reduced friction during playtesting.

On the systems side, working on persistence reinforced the importance of clear data boundaries. Separating runtime behaviour from serialised save data prevented desynchronisation issues and made debugging progression bugs far more predictable.

If I rebuilt this project today, I would formalise data models and persistence architecture earlier in development, rather than adapting them mid-production. This experience has influenced how I now structure gameplay systems in Unity.
